## Requirement Verdicts

### Traceability

Verdict: partially delivered.

The system emits categorical rubric scores, rationales, and citations, and it runs a citation verifier. The verifier is not strong enough for the requirement. It accepts any normalized substring found in a scene, which allows partial or misleading citations to pass.

Evidence:

- `backend/screenscore/pipeline/verify.py:53`
- `backend/screenscore/pipeline/verify.py:61`
- Probe result: quotes such as `cannot breathe` and `cannot breathe in this house` were accepted as verified evidence when they were only substrings of a longer line.

### Depth Architecture

Verdict: partially delivered.

The staged architecture exists: parse, segment, map pass, character pass, dimension specialists, notes, synthesis, verification, assembly. Prompt files are versioned in files, and progress is streamed. However, long-script grounding is weak because specialists mostly see a compact digest plus a few fixed excerpts, not targeted raw script evidence for all claims.

Evidence:

- `backend/screenscore/pipeline/stages.py:119`

### Formats

Verdict: partially delivered.

## Severity-Ranked Findings

### Critical: Evidence Verification Is Too Lenient

Failure scenario:

Input/state: a model cites a partial phrase such as `cannot breathe` or `cannot breathe in this house` instead of the full script line `I cannot breathe in this house anymore.`

Wrong outcome: the verifier accepts the citation as grounded because it only checks `quote in scene_text` after normalization. A fabricated or cropped quote can therefore support a rubric score.

Evidence:

- `backend/screenscore/pipeline/verify.py:53`
- `backend/screenscore/pipeline/verify.py:61`
- Probe result: six adversarial citations were checked; five were kept and one relocated, zero dropped.

Additional risk:

If the same quote appears in multiple scenes and the cited scene is missing or wrong, relocation picks the first matching scene. That can silently misattribute repeated dialogue.

### Critical: Long Feature Scripts Are Likely Under-Grounded

Failure scenario:

Input/state: a 110-page, 200-scene script.

Wrong outcome: dimension specialists must score coverage dimensions from a 20k-character digest and four excerpts. If the digest exceeds budget, quotes are dropped first, then summaries are reduced to slug-only, and finally the text is truncated. The specialist can still produce a score, but the evidence context is not enough to justify every scored claim.

Evidence:

- `backend/screenscore/pipeline/stages.py:51`
- `backend/screenscore/pipeline/digest.py:55`
- `backend/screenscore/pipeline/digest.py:62`
- `backend/screenscore/pipeline/digest.py:66`
- `backend/screenscore/pipeline/digest.py:137`

### Critical: Cache Keys Are Incomplete

Failure scenario:

Input/state: change `SCREENSCORE_NUM_CTX`, model options, engine code, parser behavior, or a dimension criteria file without bumping `PROMPT_VERSION`.

Wrong outcome: stale cached stage outputs can be reused for a new configuration, making reports unreproducible or wrong.

Evidence:

- `backend/screenscore/db/repository.py:285`
- `backend/screenscore/db/repository.py:296`
- `backend/screenscore/pipeline/stages.py:77`
- `backend/screenscore/pipeline/stages.py:81`
- `backend/screenscore/pipeline/stages.py:54`
- `backend/screenscore/pipeline/stages.py:55`

Specific problems:

- Cache key omits `num_ctx`, temperature, runtime options, engine version, parser version for pipeline parse, prompt file hashes, and dimension criteria file hashes.
- Upload parse caching uses `PARSER_VERSION`, but pipeline parse lookup uses `prompts.PROMPT_VERSION`, creating inconsistent parse cache semantics.

### Major: Parser Robustness Is Overclaimed

Failure scenario:

Input/state: a real production script with A/B scene numbering like `A12 INT. LAB - DAY` and `12B EXT. ROOF - NIGHT`.

Wrong outcome: the first scene heading is missed; later parsing collapses or misnumbers scenes, corrupting citations and scene-level coverage.

Evidence:

- `backend/screenscore/parsing/text.py:27`
- Probe result: `A12 INT. LAB - DAY ... 12B EXT. ROOF - NIGHT` produced one scene: `EXT. ROOF - NIGHT`.

Additional likely weak spots:

- OCR noise
- scanned PDFs
- dual-column dialogue from PDF extraction
- revision stars/asterisks in margins
- title pages or cast pages embedded in PDFs
- page-broken dialogue with unusual continuation markers

### Major: Graceful Degradation Is Not Actually Guaranteed

Failure scenario:

Input/state: user has a weak machine, Ollama running, but only one installed model, or no recommended reasoning model installed.

Wrong outcome: the report fails instead of degrading gracefully to a single installed small model.

Evidence:

- `backend/screenscore/pipeline/stages.py:211`
- `backend/screenscore/pipeline/stages.py:225`

### Major: Privacy Is Not Enforced at the Runtime Boundary

Failure scenario:

Input/state: `OLLAMA_URL` is set to a remote Ollama-compatible endpoint.

Wrong outcome: screenplay prompts are sent over the network during analysis, violating the "zero outbound network calls during analysis" promise.

Evidence:

- `backend/screenscore/runtime/ollama.py:24`
- `backend/screenscore/runtime/ollama.py:96`

### Major: Tests Are Too Synthetic for the Claimed Guarantees

Failure scenario:

Input/state: tests pass.

Wrong outcome: passing tests can create false confidence. The suite is fast and useful, but it does not validate feature-length analysis, real model behavior, real production PDFs, hostile citation behavior, or actual Docker startup.

Evidence:

- Test suite result: `92 passed in 1.46s`
- `backend/tests/test_pipeline.py` uses `FakeRuntime`
- `backend/screenscore/runtime/fake.py:140` returns evidence from digest quotes, which makes the verification path fit the fake
- `backend/tests/test_verify.py` covers basic fabricated quote dropping but not substring fabrications, duplicate-scene ambiguity, or quote-boundary validation

Failure scenario:

Input/state: a client checks export availability with `HEAD /api/reports/{id}/export/md`.

Wrong outcome: receives `405 Method Not Allowed` even though `GET` works.

Evidence:

### Minor: Diff Narrative Has No Contradiction Validator

Failure scenario:

Input/state: mechanical score movement says a dimension declined, but model narrative claims it improved.

Wrong outcome: UI can display contradictory comparison guidance unless the user notices the mechanical table.

Evidence:

- `backend/screenscore/pipeline/diff.py:141`
- `backend/screenscore/pipeline/diff.py:151`

## The Three Things To Fix First

1. Rebuild evidence verification around exact, anchored spans.

   Require full quote matches against parsed script lines or stored character offsets. Preserve duplicate-match ambiguity instead of silently relocating to the first scene. Reject short substrings and cropped fragments as insufficient support.

2. Add targeted retrieval for specialist passes.

   Each dimension should receive the exact scenes or lines relevant to that dimension, not only a compressed digest plus four fixed excerpts. If a score cites evidence, the model should have seen and quoted the underlying script text directly.

3. Fix cache identity.

   Cache keys should include parser version for parse, engine version, prompt file hashes, criteria file hashes, runtime options such as `num_ctx` and temperature, and any schema-affecting config. Do not rely on manually bumping one global `PROMPT_VERSION`.

## Overall Grades

### Fidelity To The Ask: C

The system implements much of the requested shape: local app, staged pipeline, local persistence, citations, diff pass, and exports. The top requirement was traceability, and that is only partially satisfied because evidence verification is mechanically too forgiving.

### Engineering Quality: B-

The code is readable and organized, with useful tests and a coherent FastAPI/React/SQLite structure. The biggest engineering weaknesses are cache correctness, parser realism, long-context grounding, and runtime/privacy enforcement.

### Honesty Of Self-Reporting: C+

The README and decisions are directionally aligned with the implementation, but they overstate evidence verification, parser robustness, graceful degradation, and privacy guarantees. The project is not lying wholesale; it is optimistic in exactly the places a real user would most depend on.

### Production Readiness: D+

This is a solid prototype and a good foundation for iteration. It is not ready for a screenwriter to trust on a script they care about, because the evidence layer can bless weak citations and the long-script path has not been proven.

## Positive Notes

- The major architectural pieces are present and inspectable.
- Cross-draft scoring blindness is structurally cleaner than many implementations: scoring and diff are separate modules.
- Local SQLite/project memory is a real implementation, not just a README promise.
- The fake runtime is clearly labeled in report metadata and the UI, which helps avoid confusing demo output with real coverage.
- The frontend report already has a useful evidence-to-script interaction path.
